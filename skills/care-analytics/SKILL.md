# CARE Analytics Query Builder

The schema catalog is bundled with this skill in `schema/` (all paths below
are relative to this skill's directory). It is sharded for progressive
disclosure — read only what the query needs.

Two directories, two owners:

- `schema/` — machine-generated by `scripts/build.sh`. Never hand-edit;
  regeneration overwrites it.
- `notes/` — human-curated context. Never touched by regeneration. This is
  where business definitions, enum literals, data quirks, and deployment
  facts live.

## Workflow

1. **Clarify before querying.** Before writing any SQL, ask the user about
   whatever is ambiguous — do not guess. Typical gaps: time window and
   timezone, facility/organization scope, what counts as an entity ("active
   encounter"? "completed visit"? admitted vs outpatient?), whether
   soft-deleted rows matter, and the desired output (single query, per-group
   breakdown, dashboard-ready view). Ask once, concisely, batching the
   questions. Skip only what the request already answers.
2. Read `schema/_conventions.md`, `schema/_index.md`, and `notes/_general.md`
   (if present) — all small, always safe.
3. Pick candidate tables from the index. If the entity or column isn't
   obvious from table names, search: `grep -ril "<keyword>" schema/tables/ notes/`
4. For each chosen table, read `schema/tables/<db_table>.md` **and**
   `notes/<db_table>.md` if it exists — notes override or extend the
   generated shard (curated enum values, semantics, known data quirks).
   Inherited columns (`id`, `external_id`, `created_date`, `deleted`,
   `meta`, …) are defined once in `schema/_base_models.md` — read it once
   per session.
5. If the query touches a JSONB column, read `schema/jsonb/<db_table>.md`
   for its shape before writing `->>` accessors. Treat `candidate_schemas`
   as strong hints, not guarantees.
6. If filtering on a `choices=` column, check `notes/<db_table>.md` first —
   curated literals may already be there. Otherwise the shard shows a symbol
   like `Status.choices`: grep the model source (path in the shard's
   `source:` header, inside the care checkout) for the literal stored
   values — never guess enum values. If the care source isn't available,
   ask the user or sample distinct values from the database.
7. Write PostgreSQL 16 SQL following `schema/_conventions.md`. Non-negotiables:
   - `WHERE deleted = false` on every BaseModel table (unless auditing deletions)
   - Join on integer `<name>_id` columns; expose `external_id` (UUID) in
     results, never the internal `id`
   - Plugin tables (`app/` sources) exist only in deployments that enable
     that plug — say so when a query depends on one

## Growing the curated notes

After answering, review what you learned that the catalog didn't tell you.
If any of it is durable, propose adding it to `notes/` and, if the user
agrees, write it there. Worth capturing:

- Enum literals you had to dig out of source or sample from the database
- JSONB shapes confirmed against real data (vs the spec hints)
- Business definitions the user clarified ("active encounter" = status IN
  (...), "revenue" = ...) — table-specific ones in `notes/<db_table>.md`,
  cross-cutting ones in `notes/_general.md`
- Data quirks (legacy rows with NULL status, columns populated only after
  date X, plugin tables absent in this deployment)

Not worth capturing: anything one-off, or already in `schema/`. Keep each
note short and dated. Format: see `notes/README.md`.

## Provenance and regeneration

`schema/PROVENANCE` records the care commit, plugin list, and generation
date this catalog was built from. To rebuild (e.g. after model changes or
with different plugins), run `scripts/build.sh` at the repo root of this
plugin — see the repo README. Rebuilds replace `schema/` wholesale and never
touch `notes/`.
